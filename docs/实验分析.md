# 实验分析

## 0. OMO / OpenCode 架构概览

下面先给出本实验中 OMO 插件在 OpenCode 上的大致工作框架。核心逻辑是：用户任务进入 OpenCode 后，由 OMO 的主编排 Agent 判断任务类型，并在需要时调用规划、执行、检索、审查等子 Agent；最终仍要产出 patch，并交给 SWE-bench harness 判定是否 resolved。

在本实验里，重点不是评估所有 OMO agent 的完整能力，而是围绕两个问题做对照：一是“带 OMO 插件是否优于不带 OMO 插件”；二是“在带 OMO 的前提下，单模型配置是否弱于多模型配置”。

## 1. 分析问题与口径

本文回答两个并行问题：

1. **问题一：带上 OMO 插件是否比不带 OMO 插件而使用单一模型更好？**
2. **问题二：带上 OMO 插件但都配置单一模型，是不是比不同模型配置要差？**

实验对象均为 SWE-bench Verified Mini 25 题正式子集(均匀采样得到)。

中间过程与最终结果主要来自以下产物：

| 类型                | 位置                                        |
| ------------------- | ------------------------------------------- |
| 逐题 metrics        | `experiments/results/runs/*.jsonl`        |
| API/token/cost 汇总 | `experiments/results/api_metrics/*.md`    |
| prediction JSONL    | `experiments/results/predictions/*.jsonl` |
| 最终 patch          | `experiments/results/runs/*/*/patch.diff` |

抽查 prediction JSONL 显示，每题包含 `instance_id`、`model_name_or_path`、`model_patch`；最终 patch 与对应 `patch.diff` 可以追溯到 harness 的 `resolved` 结果。

## 2. 问题一：带上 OMO 插件是否比不带 OMO 插件而使用单一模型更好？

这里只比较同一模型下的两种设置：

- **Baseline**：OpenCode，不启用 OMO 编排。
- **OMO Single**：OpenCode + OMO 插件，但所有 agent/category 都使用同一个模型。

### 2.1 汇总结果

| 组别                           | 题数 | Resolved | 成功率 | 延迟合计 min | 平均延迟 min | 中位延迟 min | Tokens | 平均 tokens/题 |           Cost | 平均 cost/题 |
| ------------------------------ | ---: | -------: | -----: | -----------: | -----------: | -----------: | -----: | -------------: | -------------: | -----------: |
| GPT-5.4 Baseline               |   25 |       16 |  64.0% |         44.9 |          1.8 |          1.5 |  5.94M |        237,630 |  $7.77 | $0.31 |              |
| GPT-5.4 + OMO Single           |   25 |       18 |  72.0% |        102.9 |          4.1 |          3.7 | 15.52M |        620,769 | $13.66 | $0.55 |              |
| Claude Sonnet 4.6 Baseline     |   25 |       21 |  84.0% |         68.0 |          2.7 |          1.3 | 17.79M |        711,643 | $12.38 | $0.50 |              |
| Claude Sonnet 4.6 + OMO Single |   25 |       20 |  80.0% |        132.7 |          5.3 |          3.1 | 61.57M |      2,462,940 | $34.29 | $1.37 |              |

### 2.2 GPT-5.4 组

GPT-5.4 组中，OMO Single 从 16/25 提升到 18/25，成功率从 64.0% 提升到 72.0%。但代价也明显增加：token 从 5.94M 增加到 15.52M，花费从 $7.77 增加到 $13.66，平均延迟从 1.8 分钟/题增加到 4.1 分钟/题。

- OMO 额外修复成功：`django__django-12304`、`sphinx-doc__sphinx-8551`、`sphinx-doc__sphinx-8721`。
- OMO 退化失败：`django__django-12039`。
- 共同成功：15 题；共同失败：6 题。

判断：GPT-5.4 组里，OMO Single 带来小幅成功率收益，但属于“用更多 token、成本和时间换来 +2 题”的结果。

### 2.3 Claude Sonnet 4.6 组

Claude Sonnet 4.6 组中，OMO Single 从 21/25 降到 20/25，成功率从 84.0% 降到 80.0%。同时 token 从 17.79M 增加到 61.57M，花费从 $12.38 增加到 $34.29，平均延迟从 2.7 分钟/题增加到 5.3 分钟/题。

- OMO 额外修复成功：`django__django-12276`。
- OMO 退化失败：`django__django-11964`、`sphinx-doc__sphinx-7985`。
- 共同成功：19 题；共同失败：3 题。

判断：Claude 组中，OMO Single 没有带来成功率收益，反而增加了成本和延迟。对强 Claude 单模型而言，OMO 插件的额外流程更像是额外开销，而不是稳定增益。

### 2.4 问题一结论

**带上 OMO 插件是否比不带 OMO 插件而使用单一模型更好？**

当前实验答案是：**不一定。**

- 对 GPT-5.4：OMO Single 有小幅成功率提升，16/25 -> 18/25，但成本、token 和延迟显著上升。
- 对 Claude Sonnet 4.6：OMO Single 没有提升，21/25 -> 20/25，且成本、token 和延迟显著上升。

因此，更稳妥的结论是：OMO Single 可能帮助较弱或不够稳定的模型获得少量收益，但对已经很强的 Claude Sonnet 4.6，在 SWE-bench Verified Mini 这类局部 bug 修复任务上，OMO 插件没有表现出稳定正收益。

### 2.5 模型适配解释：为什么 GPT-5.4 + OMO 上升，而 Claude + OMO 下降？

这个现象不能只用“OMO 有额外开销”解释。更关键的是：**OMO 不是一个完全模型无关的插件，它内部的 agent / category prompt 明显按模型性格分工设计。** 因此，同样是 OMO Single，把所有角色都绑定到 GPT-5.4 和把所有角色都绑定到 Claude Sonnet 4.6，得到的效果可能不同。

外部资料显示，OMO 的默认设计并不是“一个强模型跑所有角色”：

- Olares 文档说明，主模型由 OpenCode UI 选择，子 Agent 模型由 `oh-my-openagent.json` 的 `agents` 字段决定；默认配置为不同 agent 设置不同模型链，而不是所有角色共用一个模型。默认推荐中，Sisyphus/Prometheus 偏 Claude Opus，Atlas 偏 Claude Sonnet，Hephaestus/Oracle 偏 GPT-5.4，Explore/Librarian 则使用更轻量模型。来源：[Olares OMO Guide](https://www.olares.com/docs/use-cases/opencode-omo.html)
- OMO 的 agent-model matching 文档把模型分成“Communicators”和“GPT Family”：Sisyphus、Atlas 等沟通/编排型 agent 使用 Claude/Kimi/GLM 路径；Hephaestus、Oracle、Momus、`deep`、`quick` 等深度执行、审查、快速任务路径使用 GPT 路径。来源：[OMO agent-model matching](https://github.com/code-yeongyu/oh-my-openagent/blob/dev/docs/guide/agent-model-matching.md)
- 同一文档还明确说，deep specialists 是为 GPT 的 principle-driven / autonomous 风格构建的，prompt 假设模型能自主探索、目标导向执行，不建议把这些深度角色强行覆盖成 Claude。来源：[OMO agent-model matching](https://github.com/code-yeongyu/oh-my-openagent/blob/dev/docs/guide/agent-model-matching.md)
- OMO overview 也写到，Sisyphus 是 Claude-optimized prompt，但 GPT-5.4/5.5 已经有专门 prompt path；Hephaestus 则是 GPT-native autonomous agent，用于复杂调试、跨文件推理和深度执行。来源：[OMO overview](https://github.com/code-yeongyu/oh-my-openagent/blob/dev/docs/guide/overview.md)

对应到本实验，GPT-5.4 和 Claude Sonnet 4.6 的差异可以这样解释：

1. **GPT-5.4 baseline 还有“被流程帮助”的空间。** GPT-5.4 baseline 是 16/25，低于 Claude baseline 的 21/25。带上 OMO 后，GPT-5.4 获得了更强的任务结构、更多探索和更长执行预算，因此提升到 18/25。这个提升更像是 OMO harness 对相对不稳定模型的“脚手架收益”。
2. **GPT-5.4 更贴近 OMO 的 deep / reviewer / autonomous 执行路径。** OMO 的 deep、Hephaestus、Oracle、Momus 等角色本来就偏 GPT 风格。即使本实验是 OMO Single，所有 category 都映射到 GPT-5.4，也相当于让 GPT 承担了一套更适合它的“自主探索 + 原则驱动 + 审查”提示结构。
3. **Claude baseline 已经很强，OMO 反而改变了它的优势。** Claude Sonnet 4.6 baseline 已经达到 21/25，说明它在 OpenCode 原生 harness 下已经能很好地做局部定位和最小 patch。带上 OMO 后，Claude 需要处理更多 agent 角色、任务管理和长指令，容易从“直接修复 issue”转向“遵守 OMO 工作流”。
4. **把所有 OMO 角色都给 Claude 可能形成角色错配。** Claude 适合 Sisyphus/Atlas 这类沟通、规划、执行型角色，但 OMO 中的 deep/reviewer/high-category 路径并不全是 Claude-first。若 OMO Single Claude 把这些本应偏 GPT 的角色也交给 Claude，就可能让 Claude 执行不完全适配它的 prompt 风格。
5. **实验数据支持这种解释。** GPT-5.4 + OMO 是 +2 题，但 token 从 5.94M 增至 15.52M；Claude + OMO 是 -1 题，token 从 17.79M 增至 61.57M。也就是说，OMO 对 GPT 的额外开销换来了少量收益；对 Claude 的额外开销没有换来收益，反而扰动了 patch 选择。

所以，更准确的解释不是“OMO 对所有模型都好/都坏”，而是：

> OMO 更适合能从结构化 agent harness 中受益、或与其角色 prompt 风格匹配的模型。GPT-5.4 在本实验中从 OMO 的结构化流程中获得了小幅收益；Claude Sonnet 4.6 本身已经很强，而且更适合直接、局部、最小修复，因此 OMO Single 的额外角色提示和深度路径错配反而造成退化。

因此，后续报告中不应把结论写成“OMO 无效”，而应写成：**OMO 的效果具有模型适配性。对 GPT-5.4 这类在 baseline 中还有提升空间、且更适配 deep/reviewer/autonomous prompt 的模型，OMO Single 可能有收益；对 Claude Sonnet 4.6 这种 baseline 已经很强、擅长直接修复的模型，OMO Single 可能变成额外负担。**

## 3. 问题二：带上 OMO 插件但都配置单一模型，是不是比不同模型配置要差？

这里比较的是 OMO 内部两类设置：

- **OMO Single**：所有 agent/category 都使用同一个模型。
- **OMO Multi**：父 agent/executor 主要使用 Claude Sonnet 4.6，deep/reviewer/high category 配置为 GPT-5.4。

需要注意：不同模型配置只有在实际触发子 Agent 时，才算真正发生多模型协作。因此，本节把 “natural multi 未触发子 Agent” 和 “forced/reviewer 真实触发子 Agent” 分开解释。

### 3.1 25 题主结果

| 组别              | 配置                                            | 题数 | Resolved | 成功率 | GPT 子调用 | 延迟合计 min | 平均延迟 min | Tokens |   Cost |
| ----------------- | ----------------------------------------------- | ---: | -------: | -----: | ---------: | -----------: | -----------: | -----: | -----: |
| OMO Single Claude | 所有 agent/category 使用`claude-sonnet-4-6`   |   25 |       20 |  80.0% |          0 |        132.7 |          5.3 | 61.57M | $34.29 |
| OMO Multi Natural | Claude parent + GPT-5.4 high/deep/reviewer 配置 |   25 |       19 |  76.0% |          0 |         92.1 |          3.7 | 26.35M | $13.91 |
| OMO Multi Forced  | Claude parent + 强制 GPT-5.4 子 Agent           |   25 |        5 |  20.0% |        348 |        185.9 |          7.4 | 15.30M | $14.31 |

从 25 题结果看：

- Natural Multi 名义上是多模型配置，但 GPT 子调用为 0，说明它没有真正触发 GPT-5.4 子 Agent；它更像是“带多模型配置但实际由 Claude 主模型完成”的运行。
- Natural Multi 为 19/25，低于 OMO Single Claude 的 20/25，因此不能证明多模型配置优于单模型 OMO。
- Forced Multi 真实触发了 GPT 子 Agent，但成功率只有 5/25，明显低于 OMO Single Claude，说明强制前置委派会显著破坏修复效果。

Natural Multi 没有 GPT 子调用，主要原因是默认 SWE-bench 修复 prompt 并不会主动要求模型使用 `task()` / subagent。也就是说，虽然配置里把 `deep`、`ultrabrain`、`unspecified-high`、Prometheus、Metis 等路径映射到了 GPT-5.4，但如果主模型没有显式创建子任务，这些配置不会被实际调用。因此，Natural Multi 的 0 次 GPT 调用说明：**这组没有真正测试到“Claude + GPT 协作”，而是测试了“Claude 在 multi-model OMO 配置环境下独立解题”。**

### 3.2 5 题策略预检结果

这 5 题是重点挑选的诊断样本，包含 forced 失败但单模型做对的题。对同一 5 题，Claude OMO Single 是 5/5。

| 组别                    | 题数 | Resolved | 成功率 | GPT 子调用 | 延迟合计 min | 平均延迟 min | Tokens |  Cost | 说明                               |
| ----------------------- | ---: | -------: | -----: | ---------: | -----------: | -----------: | -----: | ----: | ---------------------------------- |
| Claude OMO Single       |    5 |        5 | 100.0% |          0 |         15.9 |          3.2 |  3.59M | $3.29 | 单模型 OMO 对照                    |
| Natural Multi           |    5 |        5 | 100.0% |          0 |         14.8 |          3.0 |  3.75M | $2.39 | 未触发 GPT 子 Agent                |
| Forced Delegation       |    5 |        0 |   0.0% |         67 |         36.6 |          7.3 |  2.99M | $2.62 | 强制子 Agent，诊断组               |
| Optional-on-uncertainty |    5 |        0 |   0.0% |          0 |         16.1 |          3.2 |  2.54M | $1.71 | 未触发子 Agent，但 prompt 环境改变 |
| Post-patch-review       |    5 |        0 |   0.0% |         62 |         39.9 |          8.0 |  2.81M | $2.87 | 真实触发 reviewer，但未修复成功    |

从 5 题预检看：

- Claude OMO Single 在这 5 题上是 5/5。
- Natural Multi 也是 5/5，但 GPT 子调用为 0，因此不能作为“多模型协作有效”的证据。
- Forced Delegation 和 Post-patch-review 都真实触发了 GPT-5.4 子 Agent，但均为 0/5，说明当前 multi-agent 触发方式没有改善结果，反而带来明显负效应。
- Optional-on-uncertainty 也为 0/5，且 GPT 子调用为 0，说明 prompt/策略环境本身就可能扰动 Claude 的原始修复轨迹。

#### 3.2.1 预检结果原因分析

这组 5 题的关键不是“GPT-5.4 子 Agent 能不能被调用”，而是**调用时机、任务粒度和职责边界是否适合 SWE-bench 这种局部 bug 修复任务**。OMO/Olares 文档把 OMO 定义为 OpenCode 的 multi-model orchestration plugin，主要通过 `ultrawork/ulw` 触发 Sisyphus、Hephaestus、Oracle、Atlas 等专门 Agent 协作；OMO 官方文档也把 Sisyphus 描述为面向复杂任务的编排器，把 Hephaestus/deep 定位为深度自主工作，把 Oracle/Momus 定位为架构咨询或 review 角色。这说明 OMO 的设计主场是复杂、多文件、多阶段任务，而不是每个小 bug 都默认拆给子 Agent。

外部社区实测也支持这一点。Zenn 的 ultrawork 实测记录显示，`ultrawork` pass rate 为 69.2%，低于 vanilla OpenCode 的 73.1%，请求数约为 3.5x，耗时也更长；作者给出的解释是：当任务可以放进单一上下文窗口时，orchestration overhead 会拖累结果，而 ultrawork 更适合认证迁移、大规模重构、批量功能实现等跨文件跨层任务。Rost Glukhov 的实测文章也强调，OMO 的委派不是自动发生的，通常需要显式 ultrawork 语义；它在需要外部文档、跨模块实现、长任务拆分时更可能有价值，但在简单局部修复上容易只增加 token、延迟和复杂度。

结合本实验，可以把 5 题结果拆成四类原因：

1. **Natural Multi：成功但不能证明多模型有效。** Natural Multi 是 5/5，但 GPT 子调用为 0，说明这些题实际仍由 Claude 主模型完成。它成功的原因更接近“Claude 在 OMO 环境下仍能独立修好这 5 题”，而不是“Claude + GPT 协作修好”。默认 SWE-bench prompt 并不会主动要求 `task()` / subagent，因此即使配置中存在 GPT-5.4 deep/reviewer，高阶子 Agent 也不会自然触发。
2. **Forced Delegation：前置委派造成锚定和责任拆分。** Forced Delegation 要求 Claude 在编辑前必须先调用一次 deep/reviewer。这个触发点太早，会让子 Agent 的定位、文件判断或修复建议先进入上下文，形成前置锚定。对这 5 题来说，Claude OMO Single 已经能独立 5/5；强制委派反而把一个局部修复任务拆成“先听子 Agent 判断、再由 parent 执行”的协作链，增加了误定位、过度修改和执行偏差。逐题 patch 也印证了这一点：Forced 5 题 patch 均不同于 Claude OMO Single，其中部分 Sphinx 题还触碰了额外文件或测试文件，说明它更容易偏离最小正确 patch。
3. **Optional-on-uncertainty：没有触发 GPT，但 prompt 环境已经改变。** 这组 GPT 子调用为 0，因此 0/5 不能归因于 GPT-5.4 子 Agent 质量，而应归因于 OMO multi 策略 prompt、可用 category/agent、fallback 图和运行环境对 Claude 解题轨迹的扰动。它虽然写着“只有不确定时才调用”，但 prompt 仍会让模型把注意力放到不确定性、风险、是否委派等元决策上。对 SWE-bench 的小修复，这种额外决策层可能让模型偏离 baseline/OMO Single 的直接定位路径。逐题看，Optional 有 4/5 patch 与 Claude OMO Single 不同；这说明即使没有实际子调用，策略提示本身也足以改变 patch 选择。
4. **Post-patch-review：reviewer 被调用了，但没有形成有效修正闭环。** 这组 62/62 proxy 请求成功，说明失败不是 API 或 proxy 没通，而是 reviewer 机制没有转化为正确 patch。Post-patch-review 的设计是 Claude 先修，再让 GPT-5.4 审查 diff；理论上这比强制前置委派更合理。但在本实验中，reviewer 只提供 critique，不直接拥有最终 patch，也不一定能完整复现 harness 失败条件；如果 Claude 的候选 patch 已经偏离，reviewer 未必能指出关键遗漏，甚至可能引入新的边界担忧，让 parent 进一步改坏。逐题看，Post-patch-review 部分 patch 与 Claude OMO Single 相同但仍 unresolved，说明这里还需要把 harness report 和 prediction JSONL 作为单独核查项，避免把潜在评测/运行上下文差异全部解释成模型能力差异。

因此，这 5 题更合理的解释是：**SWE-bench Verified Mini 的诊断样本本身是局部、短链路、高精度修复任务，Claude Sonnet 4.6 已经能直接完成；multi-agent 策略没有提供新增信息，反而引入了编排开销、角色提示噪声、前置锚定和 review 闭环失效。** Forced Delegation 和 Post-patch-review 应作为 forced-delegation / reviewer diagnostic；Optional-on-uncertainty 则说明即使不触发子 Agent，multi-agent prompt 环境也可能影响强单模型的原始修复轨迹。正式结论不应写成“OMO multi-agent 一定无效”，而应写成：在强 Claude + 小粒度 SWE-bench bug fix 上，当前 multi-agent 触发方式没有收益，且错误的触发点会显著降分；OMO multi 更可能适合复杂、多文件、长上下文、需要规划和外部信息聚合的任务。

### 3.3 OMO Single vs OMO Multi 的逐题变化

以 25 题 Claude OMO Single 对比 Natural Multi：

- Multi 相比 Single 新增成功：`django__django-11964`。
- Multi 相比 Single 退化失败：`django__django-12713`、`sphinx-doc__sphinx-8056`。
- 共同成功：18 题。
- 共同失败：4 题。

这说明 Natural Multi 并不是全面更强，而是出现了个别题目互换：它救回 1 题，同时丢掉 2 题，净结果低于 OMO Single。

### 3.4 问题二结论

**带上 OMO 插件但都配置单一模型，是不是比不同模型配置要差？**

当前实验答案是：**不是。至少在当前 SWE-bench Verified Mini 设置下，OMO Single 并不比 OMO Multi 差。**

- 与 Natural Multi 相比，Claude OMO Single 是 20/25，Natural Multi 是 19/25；而且 Natural Multi 没有真实触发 GPT 子 Agent。
- 与 Forced Multi 相比，Claude OMO Single 明显更好：20/25 vs 5/25。
- 在 5 题策略预检中，Claude OMO Single 是 5/5，而 forced、optional、post-patch-review 三种 multi-agent 诊断设置都是 0/5。

因此，当前多模型 OMO 配置没有证明自己优于单模型 OMO。对局部 bug 修复任务，单模型 OMO 更稳定；多模型配置只有在任务确实需要跨模块规划、检索、审查或长上下文分工时，才可能发挥优势。

## 4. 总体结论

两个问题的结论可以合并为：

1. **是否带 OMO：不一定更好。** GPT-5.4 上 OMO Single 有小幅提升，但成本和延迟明显上升；Claude 上 OMO Single 没有提升。
2. **OMO 单模型是否比多模型差：** OMO Multi 的 natural 组未真实触发 GPT 子 Agent；真实触发子 Agent 的 forced/reviewer 诊断组反而明显退化。
3. **对 SWE-bench Verified Mini 这种局部 bug 修复任务，强单模型已经很强。** OMO 的额外编排可能带来更多 token、延迟和 prompt drift，不一定带来更高成功率。
