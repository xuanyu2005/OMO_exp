# OpenCode + OMO 在 SWE-bench Verified Mini 上的小型实验报告模板

> 说明：本文档是实验报告模板。已知的实验准备信息已经预填；实验运行结果、评分表和分析结论待实验完成后补充。

## 1. 实验背景与研究问题

本实验用于调研 Oh-My-OpenAgent（OMO）的 agent orchestration 机制，并在一个小型真实代码修复任务集上验证它是否带来可观察提升。

核心研究问题：

1. 不使用 OMO、仅用单一模型完成任务时，表现如何？
2. 使用 OMO 但所有 agent 都绑定同一模型时，是否优于单模型基线？
3. 使用 OMO 且按角色配置不同模型时，是否优于 OMO 单模型配置？
4. 如果存在提升，提升主要来自任务拆解、检索、验证、代码审查、模型互补，还是其他机制？

## 3. 实验设计

### 3.1 三组对照设置

本实验至少设置三组对照。三组使用同一批 SWE-bench Verified Mini 实例、相同的任务描述、相同的测试判定标准和尽量一致的运行预算。这样可以分别观察两类因素：一是 OMO 编排机制相较普通单模型 OpenCode 的收益；二是在同样使用 OMO 的前提下，多模型角色分工是否比单模型 agent 配置更好。

| 组别          | Harness                 | 模型配置                                                                         | 目的                                                                        |
| ------------- | ----------------------- | -------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| A. Baseline   | OpenCode，无 OMO 编排   | `gpt-5.4-mini`                                                                 | 作为单模型基线，衡量普通 OpenCode 单模型完成代码修复任务的表现              |
| B. OMO Single | OpenCode + OMO Ultimate | 所有 agent/category 都使用`gpt-5.4-mini`                                       | 在模型能力不变的情况下，观察 OMO 编排、角色拆分和工作流机制本身是否带来收益 |
| C. OMO Multi  | OpenCode + OMO Ultimate | 快速检索、常规执行使用`gpt-5.4-mini`；深度分析、审查使用 `claude-sonnet-4-6` | 在同样使用 OMO 的前提下，测试多模型角色分工是否带来额外收益                 |

其中，A 组和 B 组的对比主要回答“引入 OMO 是否有效”；B 组和 C 组的对比主要回答“在已有 OMO 编排时，多模型分工是否进一步有效”。需要注意的是，A 组与 B 组之间不只差一个单独的算法模块，还可能同时包含提示词、工具组织、角色拆分和验证流程等差异。因此报告中会把 A/B 差异表述为 OMO 整体 agent harness 的收益，而不是过度归因于某一个单独机制。

### 3.2 公平性控制

为了减少混淆变量，实验过程中尽量保持以下条件一致：

1. 三组使用同一批任务实例，且每个实例都从相同的 `base_commit` 开始。
2. 三组看到相同的 `problem_statement`、`hints_text` 和测试目标信息。
3. 三组使用相同的成功标准：`FAIL_TO_PASS` 测试通过，且 `PASS_TO_PASS` 测试不出现明显回归。
4. 三组记录相同的实验指标，包括是否修复成功、目标测试结果、回归测试结果、耗时、token 或费用估算、最终 diff 质量。
5. 若 C 组因为多 agent 或多模型调用消耗更多 token 和时间，结果分析中单独报告成本与耗时，避免只用成功率评价优劣。

## 4. 当前结果极简汇总

截至 2026-07-06，Baseline / OMO Single 已完成 4 组 25 题实验；OMO Multi 已完成 3 类运行与诊断。

### 4.1 Baseline / OMO Single

| 实验组 | 模型 / 配置 | 样本 | Resolved | 成功率 | Tokens | Cost |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| Baseline | `gpt-5.4-mini` compat proxy | 25 | 16 | 64.0% | 5.94M | $7.77 |
| OMO Single | `gpt-5.4-mini` compat proxy | 25 | 18 | 72.0% | 15.52M | $13.66 |
| Baseline | `claude-sonnet-4-6` | 25 | 21 | 84.0% | 17.79M | $12.38 |
| OMO Single | `claude-sonnet-4-6` | 25 | 20 | 80.0% | 61.57M | $34.29 |

### 4.2 OMO Multi

| 实验组 | 配置 | 样本 | Resolved | 成功率 | GPT 子调用 | Tokens | Cost | 备注 |
| --- | --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| Natural Multi | Claude parent + GPT-5.4 high categories | 25 | 19 | 76.0% | 0 | 26.35M | $13.91 | 未真实触发 GPT 子 Agent |
| Forced Delegation | Claude parent + 强制 GPT-5.4 deep/reviewer | 25 | 5 | 20.0% | 348 | 15.30M | $14.31 | 仅作为 forced-delegation diagnostic |
| Policy Probe: optional | optional-on-uncertainty | 5 | 0 | 0.0% | 0 | 2.54M | $1.71 | 未触发子 Agent；不扩展到 25 |
| Policy Probe: review | post-patch-review | 5 | 0 | 0.0% | 62 | 2.81M | $2.87 | reviewer 成功调用但未提升结果 |

### 4.3 当前结论

- GPT 组：OMO Single 比 Baseline 多解 2 题，但 token 和成本明显上升。
- Claude 组：Baseline 21/25，高于 OMO Single 20/25，且成本明显更低。
- OMO Multi：natural 组未实际触发多模型；forced 组显著退化；5 题策略预检未恢复到 Claude 单模型水平。
- 当前不建议把 forced 或 policy probe 作为正式 OMO Multi 主成绩；更适合作为 multi-agent 编排扰动的诊断结果。

### 4.4 OMO Multi 机制解释

当前结果不应简单解释为“OMO multi-agent 无效”，更合理的判断是：

外部资料也支持这种解释：

- OMO 官方文档把使用方式区分为普通对话、`ultrawork/ulw` 自动执行，以及 `@plan` -> `/start-work` 的规划后执行；这说明 multi-agent 更适合复杂、多文件、多阶段任务，而不是所有小修复都强制分包。来源：[OMO Docs](https://omo.dev/ja/docs)
- OMO/Olares 文档说明：主模型由 OpenCode UI 选择，子 Agent 模型由 `~/.config/opencode/oh-my-openagent.json` 的 `agents` 字段决定，并带有 paid primary -> paid backup -> free fallback 链。来源：[Olares OMO Guide](https://www.olares.com/docs/use-cases/opencode-omo.html)
- 社区实测中，`ultrawork` pass rate 为 69.2%，vanilla OpenCode 为 73.1%，且请求数约 3.5x、耗时增加约 10 分钟；作者认为当任务能放进单模型上下文时，orchestration overhead 可能拖累效果，而 ultrawork 更适合跨文件、跨层的大任务。来源：[Zenn ultrawork guide](https://zenn.dev/lark1115/articles/ultrawork-guide-oh-my-openagent?locale=en)

结合本实验结果：

1. SWE-bench Verified Mini 多数是局部 bug 修复，强 Claude 已能在有限上下文内完成定位、修改和验证；额外 multi-agent 编排容易引入沟通成本、角色提示扰动和错误锚定。
2. forced delegation 明显退化，说明“强制前置子 Agent”不适合作为默认策略；optional-on-uncertainty 未触发子 Agent，更多反映 OMO multi prompt 环境本身对 Claude 的扰动；post-patch-review 虽触发 GPT reviewer，但未转化为有效修复提升。
3. 因此，第三实验的核心结论应表述为：在强 Claude + 小粒度 SWE-bench 修复任务上，OMO multi-agent 默认编排未带来正收益，甚至可能形成负收益；其合理使用边界更可能在跨模块、长上下文、大规模改造或多阶段工程任务中。

后续分析建议保留三类区分：正式主结果使用 Baseline 与 OMO Single；forced delegation 作为错误使用 multi-agent 的诊断组；multi-agent 是否有效则按任务复杂度、patch 文件数、失败测试复杂度和 reviewer 是否识别关键问题继续做机制分析。
