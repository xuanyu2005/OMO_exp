# 小型任务集与评分标准

## 任务选择原则

任务需要足够小，能在有限 API 预算内完成；同时不能太简单，否则无法体现 orchestration 的价值。建议每个任务限时 10-20 分钟。

## 任务列表

### T1：机制理解总结

- **输入**：阅读 Oh-My-OpenAgent README 与 orchestration 文档。
- **要求**：用 500-800 字说明 OMO 的 agent orchestration 如何工作。
- **验证**：是否覆盖规划层、执行层、worker 层、category routing、fallback/concurrency。

### T2：配置对比解释

- **输入**：阅读 `docs/examples/default.jsonc`、`coding-focused.jsonc`、`planning-focused.jsonc`。
- **要求**：总结三种配置分别适合什么任务，并说明差异原因。
- **验证**：是否准确引用 agent/category/background_task 的差异。

### T3：代码定位任务

- **输入**：询问“category 到 model 的默认映射在哪里定义？”
- **要求**：找到相关源码文件，并说明调用链。
- **验证**：是否找到 `packages/omo-opencode/src/shared/model-requirements.ts` 及 delegate-task category 相关文件。

### T4：实验设计任务

- **输入**：要求设计一个验证 OMO 编排有效性的小实验。
- **要求**：给出对照组、任务集、指标、威胁有效性分析。
- **验证**：是否包含 A/B/C 三组、任务一致性、成本/耗时控制、人工评分偏差控制。

### T5：小型 bugfix 或测试补全

- **输入**：在上游仓库中选择一个已存在测试附近的小问题，要求补充边界测试或解释如何修复。
- **要求**：完成最小改动或给出精确补丁方案。
- **验证**：相关测试通过；如果不实际修改上游，则要求输出可应用 patch。

### T6：失败模式审查

- **输入**：给定某次 agent 输出，让模型审查它可能遗漏了什么。
- **要求**：列出 3-5 个具体风险，并给出验证方法。
- **验证**：风险是否具体、可验证，是否避免泛泛而谈。

## 评分 Rubric

每个任务人工评分 1-5 分：

| 分数 | 含义 |
| ---: | --- |
| 5 | 完全满足要求，证据充分，可直接使用 |
| 4 | 基本正确，有轻微遗漏 |
| 3 | 部分正确，但遗漏关键细节或验证不足 |
| 2 | 方向相关，但存在明显错误 |
| 1 | 基本失败或无法使用 |
